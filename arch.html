设计模式原则

什么是设计模式

设计模式是个很值得思考且需要思考很深的事.
Just Do it.

六大原则

  • 单一职责原则=>Single Responsibility Principle=>SRP

一个类只干这件事且只有这个类也只干这件事,俗语:一夫一妻制

  • 里氏替换原则=>Liskov Substitution Principle=>LSP

基类出现的地方都可以完美使用其子类替换,俗语:养子防老or上阵父子兵

  • 依赖倒置原则=>Dependence Inversion Principle=>DIP

所有的实现都应该依赖于约束,无约束的实现是不可控的,其中的约束就是interface,多约束就是多接口实现,俗语:依法治国,一国两治(多约束,手动点赞)每个地方都会有地方性治安条例,在各地具体的标准/实施不一样,但是各地所解决的问题是同一个.

  • 接口隔离原则=>Interface Segregation Principle=>ISP

所有的接口都是有针对性,所要解决的问题是单一的,因为可以多实现,提高内聚性.举个栗子:国家是无数个小家组成的,家庭有收入,支出,那么收入和支出就是不同的接口,而每个家庭都必须要实现.俗语:条条大路通罗马.

  • 迪米特法则=>Law of Demeter=>LoD

低耦合，高内聚是程序设计的最终目标,低耦合就是此法则的目标,通过减少依赖或者无感知依赖达到.俗语:知其然,不知其所以然.你只要知要这个类有了它所需要的依赖,而不需要知道它怎么有的.

  • 开闭原则=>Open Close Principle=>OCP

关闭修改,开放扩展.业务的变化是一定的,但是类的修改却不是一定的,需要类在设计的时候就有横向扩展性.俗语:宪法补充案.

原则总结

具体的实现类是单一目标的,实现类能完美替换基类,类是实现的接口约束,接口约束同样也是单一目标的,约束的实现中需要借助其他约束时,在其初始化的同时注入约束的实现(其实就是依赖注入),每个约束的划分清晰,无强交集.

(以上总结以六大原则顺序而来,实质我觉得是反向的.先定义好合适约束,也就是interface需要实现的方法,以满足开闭原则.在interface或者class中使用依赖注入,以满足迪米特原则.在第一步设计约束的时候,同时也要满足接口隔离原则.class会实现所有需要实现的interface中的方法,这就是依赖倒置原则.class可以继承另一个class,继承者可以完美替换父类.每个类的实现都应该是单一需求或者功能)

tips

  • 每个repo可以创建最多20个webhook
  • 默认只订阅push事件
  • *通配符事件不建议使用
  • Payloads最大5M,不建议一次性push太多分支或者标签及备注

Webhook Headers

Name Description
X-GitHub-Event The event type that was triggered.
X-GitHub-Delivery A guid to identify the payload and event being sent.
X-Hub-Signature The value of this header is computed as the HMAC hex digest of the body, using the secret config option as the key.

soa-architecture-overview

什么是SOA

面向服务的体系结构（英语：service-oriented architecture）是构造分布式计算的应用程序的方法。它将应用程序功能作为服务发送给最终用户或者其他服务。

企业系统的架构师认为SOA能够帮助业务迅速和高效地响应变化的市场条件. 服务导向的架构在宏观（服务）上，而不是在微观上（对象）因此提高了重复使用性。同时，服务导向的架构可以简化与传统系统的互连和使用。

SOA Principle

  • 可重复使用，粒度，模组性，可组合型，物件化原件，构件化以及具交互操作性
  • 符合开放标准（通用的或者行业的）
  • 服务的识别和分类，提供和发布，监控和跟踪
