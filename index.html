设计模式原则

什么是设计模式

设计模式是个很值得思考且需要思考很深的事.
Just Do it.

六大原则

  • 单一职责原则=>Single Responsibility Principle=>SRP

一个类只干这件事且只有这个类也只干这件事,俗语:一夫一妻制

  • 里氏替换原则=>Liskov Substitution Principle=>LSP

基类出现的地方都可以完美使用其子类替换,俗语:养子防老or上阵父子兵

  • 依赖倒置原则=>Dependence Inversion Principle=>DIP

所有的实现都应该依赖于约束,无约束的实现是不可控的,其中的约束就是interface,多约束就是多接口实现,俗语:依法治国,一国两治(多约束,手动点赞)每个地方都会有地方性治安条例,在各地具体的标准/实施不一样,但是各地所解决的问题是同一个.

  • 接口隔离原则=>Interface Segregation Principle=>ISP

所有的接口都是有针对性,所要解决的问题是单一的,因为可以多实现,提高内聚性.举个栗子:国家是无数个小家组成的,家庭有收入,支出,那么收入和支出就是不同的接口,而每个家庭都必须要实现.俗语:条条大路通罗马.

  • 迪米特法则=>Law of Demeter=>LoD

低耦合，高内聚是程序设计的最终目标,低耦合就是此法则的目标,通过减少依赖或者无感知依赖达到.俗语:知其然,不知其所以然.你只要知要这个类有了它所需要的依赖,而不需要知道它怎么有的.

  • 开闭原则=>Open Close Principle=>OCP

关闭修改,开放扩展.业务的变化是一定的,但是类的修改却不是一定的,需要类在设计的时候就有横向扩展性.俗语:宪法补充案.

原则总结

具体的实现类是单一目标的,实现类能完美替换基类,类是实现的接口约束,接口约束同样也是单一目标的,约束的实现中需要借助其他约束时,在其初始化的同时注入约束的实现(其实就是依赖注入),每个约束的划分清晰,无强交集.

(以上总结以六大原则顺序而来,实质我觉得是反向的.先定义好合适约束,也就是interface需要实现的方法,以满足开闭原则.在interface或者class中使用依赖注入,以满足迪米特原则.在第一步设计约束的时候,同时也要满足接口隔离原则.class会实现所有需要实现的interface中的方法,这就是依赖倒置原则.class可以继承另一个class,继承者可以完美替换父类.每个类的实现都应该是单一需求或者功能)

官释

很多更成熟的数据库都支持预处理语句的概念。什么是预处理语句？可以把它看作是想要运行的 SQL 的一种编译过的模板，它可以使用变量参数进行定制。预处理语句可以带来两大好处：

  • 查询仅需解析（或预处理）一次，但可以用相同或不同的参数执行多次。当查询准备好后，数据库将分析、编译和优化执行该查询的计划。对于复杂的查询，此过程要花费较长的时间，如果需要以不同参数多次重复相同的查询，那么该过程将大大降低应用程序的速度。通过使用预处理语句，可以避免重复分析/编译/优化周期。简言之，预处理语句占用更少的资源，因而运行得更快。
  • 提供给预处理语句的参数不需要用引号括起来，驱动程序会自动处理。如果应用程序只使用预处理语句，可以确保不会发生SQL 注入。（然而，如果查询的其他部分是由未转义的输入来构建的，则仍存在 SQL 注入的风险）。 预处理语句如此有用，以至于它们唯一的特性是在驱动程序不支持的时PDO 将模拟处理。这样可以确保不管数据库是否具有这样的功能，都可以确保应用程序可以用相同的数据访问模式。

wiki的定义

As compared to executing SQL statements directly, prepared statements offer two main advantages:[1]

  • The overhead of compiling and optimizing the statement is incurred only once, although the statement is executed multiple times. Not all optimization can be performed at the time the prepared statement is compiled, for two reasons: the best plan may depend on the specific values of the parameters, and the best plan may change as tables and indexes change over time.[2]
  • Prepared statements are resilient against SQL injection, because parameter values, which are transmitted later using a different protocol, need not be correctly escaped. If the original statement template is not derived from external input, SQL injection cannot occur.

phpcs集成git工作流

git hook之pre-commit

在提交代码前进行代码规范检查,因此使用git hook中的pre-commit,下面是pre-commit的概念

这个钩子被 git commit 命令调用, 而且可以通过在命令中添加--no-verify 参数来跳过。这个钩子不需要参数，在得到提交消息和开始提交(commit)前被调用。如果钩子返回值不是0，那么 git commit 命令就会中止执行。

pre-commit使用

github永久链接,后续改进更新

代码实现如下

#!/bin/bash
#
# check PHP code syntax error and standard with phpcs
# author : star[github.com/star1989]
# date : 2017-02-24
PROJECT=$(git rev-parse --show-toplevel)
cd $PROJECT
SFILES=$(git diff --cached --name-only --diff-filter=ACMR HEAD | grep \\.php)
TMP_DIR=$PROJECT."/tmp"

# Determine if a file list is passed
if [ "$#" -ne 0 ]
then
    exit 0
fi
echo "Checking PHP Lint..."
for FILE in $SFILES
do
#    echo "php -l -d display_errors=0 ${FILE}"
#   echo "git show :$FILE > $TMP_DIR/$FILE"
    php -l -d display_errors=0 $FILE
    if [ $? != 0  ]
    then
        echo "Fix the error before commit."
        exit 1
    fi
    FILES="$FILES $PROJECT/$FILE"
done

if [ "$FILES" != "" ]
then
    echo "Running Code Sniffer..."

    TMP_DIR=/tmp/$(uuidgen)
    mkdir -p $TMP_DIR
    for FILE in $SFILES
    do
        mkdir -p $TMP_DIR/$(dirname $FILE)
        git show :$FILE > $TMP_DIR/$FILE
    done
    phpcs --standard=PSR2 --encoding=utf-8 -n $TMP_DIR
    PHPCS_ERROR=$?
    rm -rf $TMP_DIR
    if [ $PHPCS_ERROR != 0 ]
    then
        echo "Fix the error before commit."
        exit 1
    fi
fi

exit $?

config.m4

      AC_MSG_ERROR([example-extra lib not found. See config.log for more information.])
    ],[-L$PHP_EXAMPLE_EXTRA/lib]
    )
  fi

  PHP_NEW_EXTENSION(example, example.c $EXAMPLE_SOURCES, $ext_shared)
  PHP_SUBST(EXAMPLE_SHARED_LIBADD)

计算机编码与python编码

编码

ASCII

计算机发明时,美国人只考虑了英语,所以当时的ASCII编码只有8个比特,即1个字节.大小英文字母/数字/标点符号/特殊符号等127个字母被纳入并规范成了ASCII编码.

Unicode(Universal Multiple-Octet Coded Character Set)

ASCII编码只适用于英语国家,所以后来百家争鸣,中国有GB2312编码,韩国/日本等也都自制编码.就这样,冲突来了.
为了解决冲突,Unicode编码被协商制定出来.将世界上所有语言字符都编码成一个数字.
Unicode字符码是32位，4个字节。

UTF-8

Unicode编码虽然兼容ASCII,但是会产生极大的浪费,Double了.所以才出现了把Unicode编码转化为可变长编码的UTF-8编码.UTF-8编码氢一个Unicode字符根据不同的数字大小编码成1-6个字节.英文字母1个字节,汉字通常3字节,生僻字4-6字节.
一般情况均传输的是英文字母(都是代码).

计算机内存编码

早期Unicode编码就是16位2字节定长.但是中国文化5千年,文字太多,居然超过了65536个.So,Unicode就从定长走向了不定长的道路.
计算机处理文本的时候，内存中一般都不用UTF-8。因为UTF-8是变长编码，不从头扫描一遍，你不知道第几个字符在哪个位置上，这在处理的时候非常浪费时间。现在很多语言/程序的处理办法，是使用源于原始UTF-16的一个定长编码，只处理字符码在16位以内的字符，不支持超过16位的罕见字。这种16位定长的编码方式被称为UCS-2。那些零星的几个突破16位的字符，除非你专门研究古文或者奇怪的小语种，一般来说是遇不到的。遇到了也是黑人问号脸。
解决办法:不怕浪费空间的，就用UTF-32，不怕浪费处理时间的，就用完整的UTF-16，或者索性用UTF-8。

python编码

在python2中,'xxx'和u'xxx'是完全不一样的.前者是UTF-8编码的,也同ASCII编码,后者是Unicode编码.
Unicode转UTF-8编码:
u'aaa'.encode('utf-8')
UTF-8转Unicode编码:
'中国'.decode('utf-8')
输入中文编码前提:
# -*- coding: utf-8 -*-

在python3中,'xxx'和u'xxx'都是Unicode编码.
